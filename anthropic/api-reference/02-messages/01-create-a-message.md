# Create a Message

`POST /v1/messages`

Send a structured list of input messages with text and/or image content, and the model will generate the next message in the conversation.

The Messages API can be used for either single queries or stateless multi-turn conversations.

## Body Parameters

### Required Parameters

| Parameter | Type | Description |
|---|---|---|
| `max_tokens` | number | The maximum number of tokens to generate before stopping. Minimum: 1. Different models have different maximum values. |
| `messages` | array of MessageParam | Input messages. Our models are trained to operate on alternating user and assistant conversational turns. |
| `model` | string | The model that will complete your prompt (e.g., "claude-opus-4-6", "claude-sonnet-4-6"). |

### Optional Parameters

| Parameter | Type | Description |
|---|---|---|
| `metadata` | object | An object describing metadata about the request. Contains optional `user_id` field. |
| `stop_sequences` | array of string | Custom text sequences that will cause the model to stop generating. |
| `stream` | boolean | Whether to incrementally stream the response using server-sent events. |
| `system` | string or array | System prompt. Provides context and instructions to Claude. |
| `temperature` | number | Amount of randomness injected into the response. Ranges from 0.0 to 1.0. Default: 1.0. |
| `thinking` | object | Configuration for extended thinking. Contains `type` ("enabled" or "adaptive") and `budget_tokens`. |
| `tool_choice` | object | How the model should use the provided tools. Options: "auto", "any", "tool", "none". |
| `tools` | array | Definitions of tools that the model may use. |
| `top_k` | number | Only sample from the top K options for each subsequent token. |
| `top_p` | number | Use nucleus sampling. Recommended to alter either temperature or top_p, but not both. |
| `service_tier` | string | The service tier to use. Options: "auto", "standard_only". |
| `inference_geo` | string | Data residency control. Options: "us", "global". |
| `output_config` | object | Output configuration including format (JSON schema or text). |
| `reasoning_effort` | string | How much effort the model should put into reasoning. Options: "low", "medium", "high", "max". |
| `mcp_servers` | array | MCP servers to connect to for tool use. |

### Messages Format

Each input message must be an object with a `role` and `content`. You can specify a single user-role message, or include multiple user and assistant messages.

**Example with a single user message:**

```json
[{"role": "user", "content": "Hello, Claude"}]
```

**Example with multiple conversational turns:**

```json
[
    {"role": "user", "content": "Hello there."},
    {"role": "assistant", "content": "Hi, I'm Claude. How can I help you?"},
    {"role": "user", "content": "Can you explain LLMs in plain English?"}
]
```

**Example with a partially-filled response from Claude:**

```json
[
    {"role": "user", "content": "What's the Greek name for Sun? (A) Sol (B) Helios (C) Sun"},
    {"role": "assistant", "content": "The best answer is ("}
]
```

Each input message content may be either a single string or an array of content blocks. Using a string is shorthand for an array of one content block of type "text":

```json
{"role": "user", "content": "Hello, Claude"}
{"role": "user", "content": [{"type": "text", "text": "Hello, Claude"}]}
```

There is a limit of 100,000 messages in a single request.

### Content Block Types (Input)

- **TextBlockParam** - Text content with optional cache_control and citations
- **ImageBlockParam** - Image content from base64 or URL source
- **DocumentBlockParam** - Document content (PDF, plain text, or content blocks)
- **ToolUseBlockParam** - Tool use block with id, input, and name
- **ToolResultBlockParam** - Tool result with tool_use_id and content
- **ThinkingBlockParam** - Thinking block with signature
- **RedactedThinkingBlockParam** - Redacted thinking block
- **SearchResultBlockParam** - Search result content
- **ServerToolUseBlockParam** - Server-side tool use
- **WebSearchToolResultBlockParam** - Web search tool results
- **WebFetchToolResultBlockParam** - Web fetch tool results
- **CodeExecutionToolResultBlockParam** - Code execution tool results
- **BashCodeExecutionToolResultBlockParam** - Bash code execution results
- **ContainerUploadBlockParam** - Container upload content
- **MCPToolUseBlockParam** - MCP tool use block
- **MCPToolResultBlockParam** - MCP tool result block

## Response

Returns a `Message` object with the following fields:

| Field | Type | Description |
|---|---|---|
| `id` | string | Unique object identifier (e.g., "msg_01XFDUDYJgAACzvnptvVoYEL") |
| `type` | string | Always "message" |
| `role` | string | Always "assistant" |
| `content` | array | Array of content blocks generated by the model |
| `model` | string | The model that handled the request |
| `stop_reason` | string | The reason the model stopped generating. Values: "end_turn", "max_tokens", "stop_sequence", "tool_use", "refusal" |
| `stop_sequence` | string or null | Which custom stop sequence was generated, if any |
| `usage` | object | Token usage information |

### Usage Object

| Field | Type | Description |
|---|---|---|
| `input_tokens` | number | The number of input tokens (after last cache breakpoint) |
| `output_tokens` | number | The number of output tokens generated |
| `cache_creation_input_tokens` | number | Tokens used to create cache entries |
| `cache_read_input_tokens` | number | Tokens read from cache |
| `service_tier` | string | The service tier used |

### Content Block Types (Response)

- **TextBlock** - Text content with optional citations
- **ToolUseBlock** - Tool call with id, name, and input
- **ThinkingBlock** - Extended thinking content
- **RedactedThinkingBlock** - Redacted thinking
- **ServerToolUseBlock** - Server tool use (web_search, code_execution, etc.)
- **WebSearchToolResultBlock** - Web search results
- **CodeExecutionToolResultBlock** - Code execution results
- **MCPToolUseBlock** - MCP tool use

## Example Request

```bash
curl https://api.anthropic.com/v1/messages \
    --header "x-api-key: $ANTHROPIC_API_KEY" \
    --header "anthropic-version: 2023-06-01" \
    --header "content-type: application/json" \
    --data '{
        "model": "claude-opus-4-6",
        "max_tokens": 1024,
        "messages": [
            {"role": "user", "content": "Hello, Claude"}
        ]
    }'
```

## Example Response

```json
{
    "id": "msg_01XFDUDYJgAACzvnptvVoYEL",
    "type": "message",
    "role": "assistant",
    "content": [
        {
            "type": "text",
            "text": "Hello! How can I assist you today?"
        }
    ],
    "model": "claude-opus-4-6",
    "stop_reason": "end_turn",
    "stop_sequence": null,
    "usage": {
        "input_tokens": 12,
        "output_tokens": 8
    }
}
```

## Streaming

When `stream` is set to `true`, the response is returned as a stream of Server-Sent Events (SSE). The event types are:

- **message_start** - Contains the initial Message object
- **content_block_start** - Start of a new content block
- **content_block_delta** - Incremental update to a content block (text, tool input JSON, thinking, citations)
- **content_block_stop** - End of a content block
- **message_delta** - Message-level changes (stop_reason, usage)
- **message_stop** - End of the message stream
- **ping** - Periodic ping events
- **error** - Error event

For the complete type definitions and all nested schemas, see the API reference at https://docs.anthropic.com/en/api/messages/create.
